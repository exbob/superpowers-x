# Project Guidelines

## 如果你是一个 AI Agent

在进行任何工作前，先阅读本章的内容；并与仓库规则 `.cursor/rules/cross-compilation-workflow.mdc` 一并阅读（其中定义 Tier A/B、禁止在宿主机直接运行目标架构二进制等通用约束）。

这是一个交叉编译的嵌入式 Linux 项目，在宿主机（X86-Linux）上开发（包括编写代码、编译打包等），然后将文件部署到目标机（ARM-Linux）上运行（包括安装、测试、查看日志等）。

### 验证分工（与本项目脚本对齐）

- **Tier A（宿主机，Agent 可执行）**：以本节 **构建** 中的 `./build.sh` 为准（开发迭代优先 `./build.sh debug`），确认命令成功退出且 **`./deploy/`** 下产物符合预期；可叠加仓库内其它**仅宿主机可跑**的检查（如格式化、静态分析、主机侧单元测试）。**不要**在宿主机上直接执行目标机（ARM）的可执行文件来冒充「运行验证」。
- **Tier B（目标机 / 人）**：在目标机上安装、启动、查看日志或通过 **部署** 中的 `./deploy.sh` 将 `./deploy/` 同步到远程目标机后再验证。Agent **不得**在未收到用户粘贴的日志、输出或明确结论时，声称「已在目标机验证通过」。
- **`./deploy.sh` 边界**：若脚本依赖 SSH、密钥、网络或交互式确认，而当前环境未就绪，则由**开发者在本机**执行部署；Agent 不反复重试明显会因环境缺失而失败的远程部署。

### 规格与计划文档

若使用 `docs/superpowers/specs/`、`docs/superpowers/plans/` 等文档，其中的交叉编译命令、产物路径、部署步骤应与本节的 `./build.sh`、`./deploy/`、`./deploy.sh` **一致**，或显式引用本节，避免与仓库实际脚本漂移。

## 日志

编写代码时，日志要分多个级别，按**严重程度：** `DEBUG` < `INFO` < `WARN` < `ERROR`，区别如下：

| 级别 | 用途 | 典型场景 |
|------|------|----------|
| **DEBUG** | 详细调试信息，开发阶段使用 | 变量值、函数入参、执行流程 |
| **INFO** | 正常运行的关键节点信息 | 服务启动、配置加载、任务完成 |
| **WARN** | 异常但不影响主流程，需关注 | 重试操作、配置缺失用默认值 |
| **ERROR** | 严重错误，功能无法正常执行 | 文件打开失败、内存分配失败 |

开发时，需要开启所有日志，方便调试测试；生产环境只开启 `INFO` 及以上，避免 `DEBUG` 日志影响性能。

## 构建

软件包括 Debug 和 Release 两个版本：

- 开发阶段要构建 Debug 版本，它开启了所有日志和调试信息。
- 开发完毕后，可以构建 Release 版本，它只有 INFO 及以上级别日志，没有调试信息，编译优化更多。

使用 `./build.sh` 脚本执行构建工作。语法：

```bash
# 构建 Release 版本
./build.sh

# 构建 Debug 版本，带调试信息
./build.sh debug

# 清理
./build.sh clean
```

构建成功后产物位于 `./deploy/` 目录下，包含了所有需要部署到目标机的文件。

## 部署

使用 `./deploy.sh` 脚本把 `./deploy/` 下的所有文件发送到远程目标机上。若环境不满足自动部署条件，由开发者在已配置密钥的机器上执行该脚本，并将目标机侧现象或日志贴回会话以便继续分析。
